### §B.1 · Critical Path Rules

- All design project data → `projects/[name]/`
- All screen exports → `projects/[name]/screens/`
- Stitch project IDs → `stitch.json` only — **never hardcode elsewhere**
- `DESIGN.md` is generated by the `design-md` skill — **never write manually from scratch**
- Windows paths: use `\` in PowerShell scripts, `/` in JSON and docs

## §C · Core Workflow

### Design Project Lifecycle

```
1. BRIEF       → User describes the desired app/page
2. ENHANCE     → enhance-prompt skill polishes the generation prompt
3. SCAFFOLD    → project-scaffolder creates folder + Stitch project
4. GENERATE    → stitch-designer generates screens via StitchMCP
5. REVIEW      → design-reviewer presents result, iterates with user
6. DESIGN.MD   → design-md skill extracts design system from screens
7. LOOP        → stitch-loop autonomously builds remaining pages
8. EXPORT      → stitch-exporter saves screens locally
9. INTEGRATE   → react-components converts to React (optional)
```

### Stitch MCP Tool Reference

| Tool | Purpose |
|------|---------|
| `mcp_StitchMCP_create_project` | Create new Stitch project → returns `projectId` |
| `mcp_StitchMCP_generate_screen_from_text` | Generate screen from prompt → returns `screenId` |
| `mcp_StitchMCP_get_screen` | Retrieve screen details + screenshot URL |
| `mcp_StitchMCP_edit_screens` | Targeted edit of existing screen |
| `mcp_StitchMCP_generate_variants` | Generate design variants |
| `mcp_StitchMCP_list_projects` | List all Stitch projects |
| `mcp_StitchMCP_list_screens` | List screens in a project |

## §D · Design Standards

### Device Types
| Type | When to use |
|------|-------------|
| `DESKTOP` | Dashboards, admin panels, web apps, landing pages |
| `MOBILE` | Mobile-first apps, consumer apps, PWA designs |
| `TABLET` | Hybrid layouts when specified |
| `AGNOSTIC` | Component-level designs, UI kits |

### Model Selection
| Model | When to use |
|-------|-------------|
| `GEMINI_3_PRO` | Final, high-quality production designs |
| `GEMINI_3_FLASH` | Quick iterations, exploration, variants |

### Design Principles
- **Modern aesthetic**: Glassmorphism, gradients, dark modes where appropriate
- **Premium feel**: Avoid generic/plain color palettes
- **Typography**: Always specify modern fonts (Inter, Outfit, DM Sans, Plus Jakarta Sans)
- **Responsive**: All designs must consider breakpoints
- **Accessible**: Sufficient contrast ratios, semantic structure

## §E · Project Metadata Schema (`stitch.json`)

Each design project folder contains a `stitch.json` — the **Single Source of Truth** for all Stitch IDs:

```json
{
  "projectId": "STITCH_PROJECT_ID",
  "projectName": "Human-readable name",
  "created": "YYYY-MM-DD",
  "lastUpdated": "YYYY-MM-DD",
  "deviceType": "DESKTOP|MOBILE|TABLET|AGNOSTIC",
  "description": "Brief project description",
  "status": "active|completed|archived",
  "screens": [
    {
      "screenId": "SCREEN_ID",
      "name": "Screen name",
      "prompt": "Original generation prompt (English)",
      "exportedFile": "screens/screen-name.html",
      "lastUpdated": "YYYY-MM-DD"
    }
  ],
  "tags": ["web-app", "dashboard", "dark-mode"]
}
```

### `stitch.json` Write Rules
- **ALWAYS** update `lastUpdated` after any screen operation
- **NEVER** remove a screenId from `screens[]` without user confirmation
- **NEVER** hardcode a `projectId` anywhere other than `stitch.json`
- `status` must be one of: `active` | `completed` | `archived`

## §F · Naming Conventions

| Item | Convention | Example |
|------|-----------|---------|
| Project folders | `kebab-case` | `saas-dashboard`, `ecomm-landing` |
| Screen files | `kebab-case.html` | `hero-section.html`, `pricing-page.html` |
| `stitch.json` | Always lowercase | `stitch.json` |
| `DESIGN.md` | Always uppercase | `DESIGN.md` |
| Commits | `type(scope): msg` | `feat(stitch): add pricing screen` |
| Branch names | `feat/<kebab-name>` | `feat/saas-dashboard` |

### Commit Types & Scopes

```
Types:  feat | fix | docs | chore | design | export | refactor
Scopes: stitch | project | skill | workflow | spec | readme | assets

Good examples:
  feat(stitch): generate dashboard screen for crm-project
  design(stitch): iterate hero section — darker bg, purple accent
  export(project): save pricing screen to saas-dashboard/screens/
  docs(spec): update stitch.json schema with SITE.md field
  chore(skill): import google stitch-loop skill v1
```

## §H · Quality Gates

Before marking any design task complete:

```
☑ Screen(s) generated without error (verify via mcp_StitchMCP_get_screen)
☑ stitch.json updated — screenId + prompt + lastUpdated
☑ Design described to user (layout, colors, key components)
☑ Exported file saved to projects/[name]/screens/ (if requested)
☑ TASKS.md updated
☑ If DESIGN.md was generated — saved to projects/[name]/DESIGN.md
```

## §I · Stitch MCP Error Handling

| Error | Recovery |
|-------|---------|
| Generation timeout | Wait 2 min, call `get_screen` once. If no ID received, retry once more. **Never spam.** |
| Screen ID not found in `stitch.json` | Call `mcp_StitchMCP_list_screens` to recover it |
| Empty `output_components` | Present raw result to user, ask for guidance |
| Project not found | Call `mcp_StitchMCP_list_projects` to find correct ID and update `stitch.json` |
| `get_screen` returns no HTML | The design may need regeneration — inform user, offer options |

## §J · Project-Specific Prohibited Actions

In addition to universal prohibitions in `AGENTS.md §7`:

```
✗ Hardcode Stitch projectId or screenId anywhere except stitch.json
✗ Write stitch.json with reduced screens[] without user confirmation (data loss)
✗ Call mcp_StitchMCP_* with a projectId not read from stitch.json
✗ Write DESIGN.md manually from imagination — only via design-md skill
✗ Retry Stitch generation more than twice without user input
✗ Push to main branch directly for multi-file design changes
✗ Run update-registry.ps1 with -Force flag without verifying what changed
✗ Delete any file inside projects/ without explicit user confirmation
```

| `notebooklm` | Research & notes | Design research, spec gathering |

### Context7 Rule
When integrating Stitch exports into a web framework (Next.js, Vite, React):
**ALWAYS** call `mcp_context7_resolve-library-id` + `mcp_context7_query-docs` before writing integration code.
Never rely on training data for framework-specific API calls.
